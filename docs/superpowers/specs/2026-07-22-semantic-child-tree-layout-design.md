# Semantic Child-Tree Layout Design

**日期：** 2026-07-22

## 目标

让带有语义 `outline.parentId` 的主流程节点在自动整理时呈现出清晰的层级关系：顶层顺序节点按原有阶段/泳道规则纵向排列；子节点继承父节点的阶段和责任泳道，并在父节点右侧形成可继续向右展开的子树。同一父节点的子节点按照 `outline.order` 从上到下排列。

本次同时建立一份包含多阶段、多泳道、嵌套子节点、判断分支、跨阶段路线、资料挂靠和资料引用的复杂流程夹具，用来验证布局、阶段/泳道边界和正交连线不会互相破坏。

## 当前问题

`packages/canvas-core/src/hierarchy.ts` 的 `placeSemanticMatrix()` 当前逐个扫描语义节点，并用泳道索引决定是否换行。同一泳道里的节点会因为列重复而依次落在同一列，`outline.parentId` 只参与编号和教程派生，没有参与几何布局。因此 `2.1.1`、`2.1.2` 和 `2.1.2.1` 看起来像普通顺序节点，无法从画面上区分父子关系。

当前代码还包含资料附录外置、阶段边界和语义连线路由的改造。新的布局必须在这些已存在的几何约束上工作，而不是重新引入旧的固定泳道网格。

## 设计边界

### 保留的行为

- 没有语义字段的旧文档继续使用现有拓扑布局和兼容的资料定位逻辑。
- 顶层节点的阶段顺序、泳道顺序、同泳道纵向排列、跨泳道向右同排和向左换行规则继续有效。
- `CanvasDocument` 的稳定节点 ID、边 ID、`outline`、`attachment` 和 `semantic` 数据协议不变。
- `routeCanvasEdges()` 不改变业务边关系；它只读取自动整理后的节点矩形重新计算派生路线。
- 资料附录仍位于阶段框外侧，不占用主流程节点的泳道列，也不生成资料引用的物理连线。

### 明确不做的事情

- 不为每一层子节点创建新的 `FlowLane`。
- 不根据旧坐标反推新的父子顺序。
- 不改变节点创建、拖拽、删除、撤销、发布或数据库协议。
- 不把作者明确设置到其他阶段/泳道的子节点静默改回父节点的阶段/泳道。

## 布局模型

### 1. 语义块和可内联子节点

自动整理在已有 `hasSemanticFlow(document)` 分支内构建 `SemanticLayoutBlock`。每个块从一个尚未布局的语义主节点开始，并递归吸收满足以下条件的子节点：

1. 子节点的 `outline.parentId` 指向当前块中的父节点；
2. 子节点和父节点归属于相同的有效阶段和有效泳道；
3. 子节点尚未被另一个块吸收。

吸收后的块是一个树，而不是扁平列表。子节点的显示顺序使用语义树已经确定的兄弟顺序，即 `outline.order`、文档索引和稳定 ID 的确定性排序。

如果作者明确把子节点改到其他阶段或泳道，该节点不进入父节点的内联树，而是作为其真实阶段/泳道中的独立布局块参与矩阵排布。这样既保留父子语义，又不让画面上的责任泳道与数据冲突。

### 2. 泳道内部的动态子列

每个泳道仍然只有一个责任归属，但可以拥有多个几何子列：

```text
泳道「业务」
┌──────────┬──────────┬──────────┐
│ 顶层节点 │ 子节点列 │ 深层子列 │
│ A        │ A.1      │ A.1.1    │
│ B        │ A.2      │ A.2.1    │
└──────────┴──────────┴──────────┘
```

子列宽度取该泳道对应深度上所有节点的最大宽度，并加固定的子列间距。泳道的总宽度因此由该泳道最大子树深度决定；后续泳道的 `x` 坐标从扩展后的前一泳道右边开始计算，不会被子树穿透或覆盖。

同一个泳道中的不同顶层块共用这些子列基线，因此父节点和它的子节点在视觉上形成稳定的垂直列，阶段或泳道变更不会依赖旧坐标。

### 3. 子树的垂直排列

每个布局块递归计算 `subtreeHeight`：

- 节点本身占用自己的高度；
- 子节点从父节点的 `y` 开始向下堆叠；
- 相邻子树之间使用 `NODE_GAP_Y`；
- 块高度是节点自身高度和子树堆叠高度的最大值。

因此：

- `A.1`、`A.2` 一定在同一子列按顺序从上到下；
- `A.2.1` 会在 `A.2` 的右侧继续向下展开；
- 下一个顶层节点必须从整个前一个布局块的底部之后开始，不能覆盖前一个块的任何后代。

阶段内部仍然只用顶层块决定“同一行 / 换行”：泳道向右推进可以同排，回到当前或更左泳道就换到下一行。每一行的高度取其中所有块的最大 `subtreeHeight`，再叠加该行的资料附录轨道预留高度。

### 4. 阶段框和泳道框

- 阶段框继续由最终位置上的主流程节点计算，因此会包住所有子列，但不会被现代资料附录卡片撑宽或撑高。
- `getSwimlaneBounds()` 在语义布局下使用动态泳道列宽，确保泳道框包住该泳道的所有子节点。
- 没有节点的已配置阶段/泳道仍然保留最小可见尺寸。
- 未分阶段或未分泳道的节点继续进入诊断区域，不会改变有归属泳道的列基线。

### 5. 资料附录和连线

布局完成后再执行现有 `placeContent()`：

- 附件 owner 使用新的阶段边界判断左侧或右侧附录轨道；
- 附录组的垂直预留根据最终子树行高计算，避免覆盖后续主节点；
- `RESOURCE_REFERENCE` 不加入物理路线；
- 普通流、判断分支、异常和回流仍交给 `routeCanvasEdges()`，节点矩形变化只会触发安全路线重算。

## 复杂测试流程图

测试夹具 `complex-semantic-flow` 使用三个阶段和四个泳道：

| 阶段 | 主要泳道 | 语义内容 |
| --- | --- | --- |
| 需求确认 | 业务、ERP | `1` 接收需求、`2` 整理需求树、`2.1` 标准化、`2.1.1` 解析、`2.1.2` 人工修订、`2.1.2.1` 复核、`2.2` 确认、`3` 判断是否通过、`3.B1` 通过、`3.B2` 驳回、`3.B2.1` 修订 |
| 生产准备 | ERP、工艺员 | `4` 排产、`4.1` 锁定资源、`5` 打样、`5.1` 检验 |
| 交付 | 质检、业务 | `6` 出货、`6.1` 出货复核、`7` 关闭 |

它还包含：

- `2`、`3`、`6` 的主挂靠 Markdown/图片/视频资料；
- 从 `3.B2.1` 指向 `2.R1` 的 `RESOURCE_REFERENCE`；
- `3.B2 → 2` 的异常回流；
- `6 → 3` 的重试回流；
- `3.B1 → 4`、`4 → 5`、`5 → 6` 的跨泳道/跨阶段边；
- 一个展开子指南来源节点，用来确认 source-derived 节点不参与主流程列宽；
- 随机旧坐标和不同节点尺寸，用来证明布局不读取历史位置且不会依赖固定卡片大小。

回归断言分为四层：

1. **树几何：** 子节点 `x` 大于父节点 `x`；同胞节点 `y` 严格按语义顺序递增；深层子节点继续右移；顶层块和后续顶层节点无矩形重叠。
2. **容器几何：** 每个主节点落在对应阶段框和泳道框内；泳道框之间不重叠；阶段框不包含现代资料附录卡片。
3. **边和资料：** 普通流、分支、异常、重试都得到路线；资料引用没有路线；附件仍在阶段外侧，并且不会覆盖主节点。
4. **兼容性：** 现有旧文档、旧拓扑布局、阶段移动、资料移动、路由和性能测试全部保持通过。

## 验证策略

1. 先在 `hierarchy.test.ts` 写复杂夹具的失败断言，确认当前实现确实会把子节点留在父节点同列。
2. 以最小递归布局实现使 canvas-core 定向测试通过，再清理重复计算。
3. 运行 canvas-core 全量测试、类型检查和构建；同步运行 Web 的 GuideEditor 定向测试和类型检查。
4. 在独立开发端口启动当前 worktree 的真实编辑器，打开复杂流程夹具或等价演示文档，检查 DOM 矩形、阶段/泳道边界、SVG 路线和浏览器控制台。
5. 最终运行 `pnpm test`、`pnpm typecheck`、`pnpm build`、`git diff --check`，并确认没有把主工作区的无关文件纳入本次变更。

## 风险与回退

- **泳道宽度变大：** 这是子树可读性所需的显式空间成本；通过按最大深度扩展单个泳道，避免改变泳道数量和顺序。
- **子树很深：** 阶段会自然变高，后续阶段向下错开；不会压缩节点到不可读的间距。
- **显式跨泳道子节点：** 作为真实泳道中的独立块布局，避免错误继承；其父子关系仍保留在语义树中。
- **语义字段异常或循环：** 继续使用 `deriveSemanticFlow()` 的确定性回退，不让布局递归无限展开。
- **新布局出现回归：** `hasSemanticFlow()` 是唯一开关；移除本次语义分支修改即可恢复旧文档路径，且不会改动持久化数据。
