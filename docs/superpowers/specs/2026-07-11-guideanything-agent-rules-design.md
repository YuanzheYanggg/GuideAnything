# GuideAnything 项目级 Agent 规则设计

## 目标

在仓库根目录新增 `AGENTS.md`，为后续进入 GuideAnything 的 Codex 任务提供可自动发现、可执行的项目约束。重点防止测试或开发命令被中断后遗留高 CPU Node/Vitest 进程，同时保持现有前后端、数据和验证边界清晰。

## 采用方案

仅在仓库根目录维护一份 `AGENTS.md`。根目录规则覆盖整个 monorepo，避免与 `README.md` 重复；README 继续面向开发者说明产品和启动方式，AGENTS 文件专门约束自动化开发行为。

## 内容结构

`AGENTS.md` 包含以下部分：

1. 项目概览：说明 Web、API、共享协议、画布算法和本地数据目录的职责。
2. 工作原则：先读现有文档与修改，保留用户改动，优先最小安全变更。
3. 常用命令：记录安装、启动、数据库和分层验证命令。
4. 验证要求：关键修改至少运行最小相关测试；完成前执行与风险匹配的类型检查、测试或构建。
5. 进程生命周期：规定测试和开发服务器必须受控运行，并在中断、超时或失败后清理完整进程树。
6. 安全边界：只结束已确认属于 GuideAnything 的进程，不处理 `kernel_task` 或无关监听器，不自动重置数据。

## 进程生命周期规则

- 启动 `pnpm dev`、Vitest、Playwright 或其他可能派生子进程的命令时，保留可追踪的会话或独立进程组。
- 同一验证命令仍在运行时不得重复启动；先读取输出并判断是正常运行、失败还是卡住。
- 命令超时、被中断或不再需要时，先向完整的 `pnpm -> Vitest/Playwright -> worker` 进程树发送 `TERM`，短暂等待后仅对仍存活且已确认属于本项目的 PID 使用 `KILL`。
- 测试结束后检查退出码，并核对没有以 GuideAnything 为工作目录的 Vitest、Vite、tsx 或 Playwright 残留进程。
- 清理服务器前先映射 `5173` 和 `3001` 的监听 PID及其工作目录；只结束属于本仓库的监听器。
- `kernel_task` 是 macOS 内核进程，永远不应尝试停止。
- 进程异常必须先记录命令、PID、父进程、工作目录和监听端口，再判断根因；不得靠进程名称批量清理所有 Node 进程。

## 验收方式

- `AGENTS.md` 位于仓库根目录，并明确适用于整个 monorepo。
- 文档中的命令与根 `package.json`、应用端口及现有 README 一致。
- 进程规则覆盖受控启动、重复启动防护、超时/中断清理、退出后残留检查和无关进程保护。
- 文档不要求自动部署、付费资源、外部发布或破坏性数据库操作。

## 非目标

- 不修改应用代码、测试代码或 package scripts。
- 不把本地机器的固定 PID 写入规则。
- 不承诺所有平台使用同一种进程清理命令；规则描述必须达到的结果，并允许按 macOS/Linux 环境选择安全实现。
