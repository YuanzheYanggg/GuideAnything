# 草稿历史变更提示设计

## 目标

让用户在恢复草稿前知道某个 revision 相对上一版改动了什么，而不是只看到“无摘要”。

## 已确认范围

- 历史接口返回由服务端生成的 `changeSummary`。
- 变更摘要基于相邻的不可变历史快照计算，不把指南当前摘要误当成版本变更说明。
- 摘要覆盖指南元数据、节点、连线和教学步骤；节点变更尽量显示节点名称。
- 历史列表卡片和“确认恢复草稿”对话框都显示该摘要。
- 没有可比较的上一版时显示“初始草稿”；无法检测到结构差异时显示明确的无差异提示。
- 不修改历史表结构，也不向前端返回完整历史文档；服务端只返回摘要字符串。

## 数据流

`guide_draft_revisions.draft_document_json` -> API 读取相邻快照 -> `changeSummary` -> `GuideDraftHistorySnapshotSchema` -> `DraftHistoryDialog`。

历史结果按保存时间倒序排列。每一项与列表中的下一项（更旧 revision）比较；最旧项没有比较基准，因此标记为初始草稿。

## 变更摘要规则

- 指南标题、摘要、标签变化分别显示元数据提示。
- 节点按 id 判断新增、删除、更新，并在可用时显示节点名称。
- 连线和教学步骤显示新增、删除、更新数量。
- 多项变更使用短句和 `·` 分隔，避免卡片高度失控。
- 旧历史文档解析失败时保持接口可用，返回“暂无可比较的变更说明”。

## 错误与兼容性

- 历史文档仍由已有 `CanvasDocumentSchema` 校验；摘要计算失败不能阻止历史列表载入。
- `changeSummary` 是服务端返回的必填字段，前端不再把空的指南摘要作为变更提示。

## 验证

- API 集成测试验证历史列表返回具体变更摘要且不泄露 `nodes` 文档。
- Web 组件测试验证历史卡片和恢复确认框展示摘要。
- Web 全量测试、类型检查、lint、构建和 `git diff --check` 必须通过。
