# GuideAnything 语义流程矩阵设计

## 目标

把“阶段、泳道、顺序、从属、分支、资料归属”存为画布的业务语义；节点坐标、教程目录、资料附录和连线全部由这些语义生成。自动整理不再读取作者开始时的拖拽坐标来猜业务关系。

## 已确认的产品规则

1. 真实流程节点必须同时有阶段和责任泳道；阶段按 `stage.order` 自上而下，泳道按 `lane.order` 自左而右。未选归属只允许作为编辑中的待归类状态，不能成为自动整理后的正式节点。
2. 一级流程号全局连续，不按阶段或泳道重置：`1`、`2`、`3`。
3. 普通从属步骤用纯数字：`5.1`、`5.2`、`5.2.1`。
4. 判断分支用 `B`：`6.B1`、`6.B2`；分支含义由边标签表示，例如“通过”“不通过”。
5. 资料用 `R`：`6.R1`、`6.R2`；资料不占据业务流程号，也不参与主流程排位。
6. 一份资料有一个主挂靠节点；其他节点只保留可跳转的资料引用，不复制资料卡、也不在画布重复画线。
7. 顶部“开始 / 流程 / 判断 / 数据 / Markdown / 图片 / 视频”菜单保留，仍只表达节点类型。普通节点创建默认是同级后续步骤；只有选中节点后的“添加子步骤”创建 `N.1`。
8. 从普通节点拉线到空白处创建真实节点时，继承来源节点的阶段和泳道，创建同级后续步骤；从判断节点拉线时创建 `N.B#`；从任意主节点创建资料时创建 `N.R#`。
9. 修改流程顺序只能通过“插入前 / 后”“添加子步骤”“移动到此节点后”或结构目录的排序动作完成。拖动画布节点不能改变编号顺序；拖入另一阶段或泳道必须明确确认归属改变，再重新整理。
10. 教程不再有独立的人工 `steps[]` 编辑路径。教程目录由编号树按“主节点、它的资料、它的子步骤”的稳定顺序派生；旧 `steps[]` 只用于旧文档兼容回退。

## 数据模型

保持 `schemaVersion: 1`，新增字段全部可选，避免旧草稿和已发布版本无法读取。

```ts
type CanvasNodeBase = {
  outline?: {
    parentId?: string;
    order: number;
    kind: 'STEP' | 'BRANCH';
  };
  attachment?: {
    ownerNodeId: string;
    order: number;
  };
};

type CanvasEdge = {
  semantic?: {
    kind: 'FLOW' | 'BRANCH' | 'EXCEPTION' | 'RETRY' | 'RESOURCE_REFERENCE';
    order?: number;
  };
};
```

- `outline` 只允许编辑态一级主流程节点使用。`parentId` 缺失表示全局一级步骤；`kind: 'BRANCH'` 只允许父节点是判断节点。
- `attachment` 只允许 Markdown、图片和视频使用，且 `ownerNodeId` 必须是一个编辑态一级主流程节点。旧 `contentParentId` 继续读取；当新 `attachment` 缺失时，它是主挂靠的兼容来源。
- `RESOURCE_REFERENCE` 是不可见语义边，用于记录一个主节点引用别处的资料。它不能成为自动布局骨架、物理连线或重复资料卡。
- 现有无 `semantic` 的主节点边读取为 `FLOW`。判断节点的旧 `yes/no` handle 在未迁移时仅作为 `BRANCH` 顺序的兼容回退，不再是未来的业务真相。

## 语义编号与教程派生

`deriveSemanticFlow(document)` 返回不写入文档的 `SemanticFlow`：

```ts
type SemanticItem = {
  nodeId: string;
  code: string;
  kind: 'STEP' | 'BRANCH' | 'RESOURCE';
  parentId?: string;
  order: number;
};
```

编号由每个父级下的 `order` 重新压实得到；插入节点不改变既有相对次序，但会使后续显示编号改变。稳定节点 ID、边 ID、评论和引用永远使用原 ID，而不使用显示编号。

学习路径从该树预序遍历派生：先显示主节点，再显示它主挂靠的资料，再显示其普通子节点或分支节点。共享资料只在主挂靠处出现一次；引用节点跳转到该资料的主挂靠位置。

## 自动矩阵布局

自动整理的输入是 `SemanticFlow`，不是旧 `position`：

1. 固定阶段横带和泳道纵列。无归属主节点保留在“待归类”诊断区，不作为正式矩阵格。
2. 按语义编号扫描每个阶段中的主节点。若下一节点位于当前或更右的泳道列，可放在同一流程行；若回到左侧泳道或当前格已占用，则开始下一流程行。因此阅读顺序固定为“从左到右，再从上到下”。
3. 判断分支可在同一流程行按 `B#` 并列；汇合后的一级节点从下一可用流程行继续。异常和返工不改变编号树。
4. 每个主节点的主资料形成一组黄色附录框，组内按 `R#` 排列，资料不再进入流程 rank。主节点只到附录组画一根展示连线；引用资料没有画布连线。
5. 路由读取边的 `semantic.kind`：`FLOW` 为主路径，`BRANCH` 为分支，`EXCEPTION` 与 `RETRY` 走虚线外侧通道，`RESOURCE_REFERENCE` 不渲染。只有手动路由模式才保留作者明确编辑过的路径。
6. 自动整理预览必须列出未分阶段、未分泳道、未挂靠资料、多主挂靠候选、循环、重排编号的数量与逐项编号变化；应用后作为一条可撤销历史记录。

## 编辑器交互

- 顶部菜单不新增常驻的“后续 / 子步骤 / 资料”按钮。
- 选中节点时，普通顶部创建按“同级后续步骤”处理并继承阶段、泳道；没有选中节点时创建待归类一级步骤。
- 拉线弹出菜单按来源类型确定关系：普通节点是 `FLOW` 后续步骤，判断节点是 `BRANCH`，资料是主挂靠资料。
- 节点快捷菜单和结构目录提供“在前插入”“在后插入”“添加子步骤”“移动到此节点后”“添加资料引用”。只有这些动作会改变 `outline`。
- 结构目录以全局编号树显示节点和资料，并给资料引用显示可点击的 `↗ 引用 6.R1`。点击后选中、聚焦主挂靠资料。
- 顶部“自动整理”保留预览 / 应用 / 取消；预览期间不持久化任何语义改动。

## 兼容与边界

- 不静默依据节点坐标推断新的业务顺序。
- 对旧文档，编号回退使用入口与主流程边的稳定拓扑顺序，再使用文档数组顺序打破并列；旧 `steps[]` 只在没有任何语义顺序时作为教程回退。
- 有一个主流程边的未挂靠旧资料，在预览中作为“可自动主挂靠”的候选；有多个候选时留在待整理区并要求作者选择。
- 不改数据库表或版本发布协议；文档 JSON 保持向后兼容。
- 本期不做协同排序、跨指南资料复用或基于 AI 自动推断阶段/泳道。

## 验收

1. 契约拒绝无效父级、无效资源主挂靠、非判断节点的 `BRANCH` 和非法引用边；旧文档仍可解析。
2. 纯函数测试覆盖全局一级编号、普通子步骤、判断分支、资料编号、共享资料引用、插入重排与旧文档回退。
3. 矩阵布局测试证明阶段横带、泳道纵列、向右同排 / 向左换行、资料不占流程格，以及异常/返工不影响编号。
4. 编辑器测试证明顶部菜单和拉线菜单创建的默认关系、子步骤专用入口、排序预览、阶段/泳道变更确认、资料引用跳转和撤销。
5. 学习模式和知识快照测试证明教程由语义树派生，资料只出现一次且可从引用处跳回主挂靠。
6. 最终通过相关测试、全量 `pnpm test`、`pnpm typecheck`、`pnpm build`、`git diff --check`，并以真实浏览器检查自动整理后的矩阵、黄色资料附录和引用跳转。
